Tab Suspender for Chrome in 2026: What to Check Under Manifest V3

6-minute read · Reviewed July 2026

Modern Chrome tab managers should be judged by what they do with your tabs, which permissions they request, and how clearly they protect active work. Manifest V3 changes an extension's background architecture, but it does not by itself prove that a tab manager is safe or reliable.

What Manifest V3 changes

Manifest V3 extensions use service workers for background event handling instead of long-lived background pages. Chrome normally stops an extension service worker after a period of inactivity and starts it again when an event needs handling.

That means a well-designed tab manager must store important state rather than relying on in-memory global variables. It should also recover cleanly after Chrome restarts or the service worker wakes again.

The modern approach: chrome.tabs.discard()

Chrome provides chrome.tabs.discard() to extensions. When Chrome accepts the request, it:

  • Discards the tab from memory
  • Keeps the tab visible in the tab strip
  • Reloads the page when the tab is activated again

The API will not discard an active or already discarded tab, and a request can fail if Chrome has no eligible tab. An extension should therefore show which tabs are eligible and report the result instead of promising that every tab will sleep.

What to look for in a 2026 tab suspender

  • A visible review step: you should know which tabs will be affected before an action runs.
  • Clear protections: active, pinned, audible, local, and explicitly protected tabs should not be selected by default.
  • Recoverable state: workspaces and settings should survive service-worker restarts.
  • Purpose-limited permissions: the store listing should explain why tab, storage, alarm, group, and host access are needed.
  • Honest estimates: memory savings vary by page and system; a fixed per-tab promise is not credible.
